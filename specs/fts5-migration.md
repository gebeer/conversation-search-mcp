# FTS5 Migration Spec

Replace the in-memory `bm25s` search backend with SQLite FTS5 for persistent indexing,
richer query syntax, and reduced dependencies.

> **Revision note (2026-03-10):** Revised after peer review with Codex (GPT-5.4).
> Changes: query escaping strategy, SQL-level filtering, threading model,
> schema versioning, FTS5 availability check, snippet markers, pattern-scoped
> stale deletion. See §Search, §Indexing Strategy, §Error Handling, §Schema.

## Motivation

Comparison with [arjunkmrm/recall](https://github.com/arjunkmrm/recall) surfaced a better
search backend choice. recall uses SQLite FTS5 (stdlib, zero dependencies) while we use
`bm25s` (pip dependency, in-memory only). FTS5 is strictly better for our use case:

| Capability | bm25s (current) | SQLite FTS5 (target) |
|------------|-----------------|---------------------|
| Persistence | None (rebuild every start) | On-disk, survives restarts |
| Query syntax | Keywords only | Phrases, boolean, prefix, NEAR |
| Dependencies | `bm25s` (pip) | `sqlite3` (stdlib) |
| Snippet extraction | First 300 chars truncated | Highlighted context windows |
| Recency bias | None | Configurable time-decay boost |
| Memory footprint | ~250 MB in-memory index | Minimal (SQLite handles paging) |
| Stemming | Via bm25s tokenizer | Porter stemmer built-in |

## What Changes

**Backend swap.** The `ConversationIndex` class internals change from in-memory arrays +
bm25s retriever to SQLite FTS5 queries. The public interface stays identical.

**What stays the same:**
- MCP server architecture (FastMCP, stdio/SSE)
- 4-tool API (search, list, read_turn, read_conversation)
- Turn-based indexing (user + assistant paired as semantic units)
- Full-fidelity reads via JSONL re-parse
- Daemon mode with auto-spawn and idle timeout
- Filesystem watchers (watchdog) for live reindex
- CLI subcommands (search, list, read-turn, read-conv)

**What changes:**
- Search backend: bm25s → SQLite FTS5
- Index storage: RAM → disk (`~/.cache/conversation-search/index.db`)
- Query behavior: implicit OR → implicit AND (FTS5 default)
- Snippets: truncated first-300-chars → highlighted context windows
- New: recency boost (30-day half-life)
- New: FTS5 query syntax (phrases, boolean, prefix)
- Dropped dependency: `bm25s`

## Schema

```sql
-- Session metadata for incremental indexing and list_conversations
CREATE TABLE IF NOT EXISTS sessions (
    session_id   TEXT PRIMARY KEY,
    file_path    TEXT NOT NULL,
    project      TEXT NOT NULL DEFAULT '',
    slug         TEXT NOT NULL DEFAULT '',
    summary      TEXT NOT NULL DEFAULT '',
    cwd          TEXT NOT NULL DEFAULT '',
    git_branch   TEXT NOT NULL DEFAULT '',
    first_ts     TEXT NOT NULL DEFAULT '',  -- ISO 8601
    last_ts      TEXT NOT NULL DEFAULT '',  -- ISO 8601
    turn_count   INTEGER NOT NULL DEFAULT 0,
    mtime        REAL NOT NULL DEFAULT 0,   -- file mtime for cache
    size         INTEGER NOT NULL DEFAULT 0  -- file size for cache
);

-- FTS5 full-text index over turns
CREATE VIRTUAL TABLE IF NOT EXISTS turns_fts USING fts5(
    text,                    -- combined: user_text + assistant_text + tool names
    session_id UNINDEXED,
    turn_number UNINDEXED,
    project UNINDEXED,
    timestamp UNINDEXED,     -- ISO 8601, for recency calculation
    tokenize='porter unicode61'
);
```

**Design notes:**
- `text` is the only indexed column — combined user + assistant + tool names, same as current.
  FTS5 searches this column. Metadata columns are UNINDEXED (stored but not searchable).
- `sessions` table stores file mtime/size for incremental indexing (same cache strategy as current).
- No content-external table — regular FTS5 is simpler and our scale (~3000 turns) doesn't need it.
- Separate `sessions` table (not FTS5) for fast metadata queries in `list_conversations`.

**Schema versioning:**
- Use `PRAGMA user_version` to track schema version (start at 1).
- On startup, check `user_version`. If it doesn't match the expected version,
  drop all tables and rebuild from scratch. Increment version on any schema change.
- This is safe because the DB is a cache — JSONL source files are the source of truth.

**Assumptions (documented invariants):**
- `session_id` is a UUID generated by Claude CLI. Uniqueness is guaranteed by the source.
  Duplicate/copied transcripts sharing the same UUID are undefined behavior.
- JSONL files are append-only (Claude CLI appends turns). The mtime+size cache key
  is valid under this invariant — same-size rewrites do not occur.

## Indexing Strategy

Same incremental pattern as current, but writing to SQLite instead of in-memory arrays.

```python
def build(self, pattern: str) -> None:
    """Incremental index build. Only reparses changed files."""
    conn = self._get_connection()

    # 1. Discover all JSONL files matching pattern
    # 2. For each file:
    #    a. Check mtime + size against sessions table
    #    b. If unchanged: skip
    #    c. If changed or new:
    #       - Delete old turns: DELETE FROM turns_fts WHERE session_id = ?
    #       - Delete old session: DELETE FROM sessions WHERE session_id = ?
    #       - Parse JSONL into turns (reuse existing _parse_conversation)
    #       - INSERT into turns_fts and sessions
    # 3. Remove stale sessions (pattern-scoped deletion):
    #    Only delete rows whose file_path matches the current glob pattern's
    #    scope AND is not in seen_paths. Do NOT delete rows from other patterns.
    # 4. PRAGMA optimize (periodic FTS5 maintenance)
```

**Cold start** (empty DB): Full index build. With 1253 files / 383 MB of JSONL, expect
~5-10 seconds (dominated by JSON parsing, not SQLite writes).

**Warm start** (existing DB): Only reparses files with changed mtime/size. Typically
0-2 files changed → sub-second.

**Daemon restart**: Opens existing DB, runs incremental build. No full rebuild needed.
This is the key persistence win over bm25s.

## Search

### BM25 + Recency Boost

FTS5 provides BM25 ranking via `ORDER BY rank` (built-in, no extra code).

Add a recency boost using recall's formula:

```python
def search(self, query, limit=10, session_id=None, project=None):
    # Build query with SQL-level filtering (not post-filter)
    conditions = ["turns_fts MATCH ?"]
    params: list = [query]

    if session_id:
        conditions.append("session_id = ?")
        params.append(session_id)
    if project:
        conditions.append("project LIKE ?")
        params.append(f"%{project}%")

    where = " AND ".join(conditions)

    # Over-fetch for recency reranking
    candidate_limit = limit * 3
    params.append(candidate_limit)

    sql = f"""
        SELECT
            text, session_id, turn_number, project, timestamp,
            rank,
            snippet(turns_fts, 0, '[[', ']]', '...', 24)
        FROM turns_fts
        WHERE {where}
        ORDER BY rank
        LIMIT ?
    """
    rows = conn.execute(sql, params).fetchall()

    # Separate COUNT query for accurate total
    count_sql = f"SELECT COUNT(*) FROM turns_fts WHERE {where}"
    total = conn.execute(count_sql, params[:-1]).fetchone()[0]

    results = []
    now = time.time()
    for row in rows:
        text, sid, turn_num, proj, ts, bm25_rank, snippet = row

        # Recency boost: 20% weight, 30-day half-life
        # FTS5 rank is negative (more negative = better match), so negate
        # first to get a positive score, then boost recent results.
        age_days = _age_in_days(ts, now)
        recency = math.exp(-0.693 * age_days / 30)
        bm25_score = -bm25_rank  # positive, higher = better
        blended = bm25_score * (1 + 0.2 * recency)

        results.append({
            "session_id": sid,
            "project": proj,
            "turn_number": turn_num,
            "score": round(blended, 4),
            "snippet": snippet,
            "timestamp": ts,
        })

    # Re-sort by blended score (critical: must happen BEFORE trimming to limit)
    results.sort(key=lambda r: -r["score"])
    return {"results": results[:limit], "query": query, "total": total}
```

### FTS5 Query Syntax

Exposed directly to users. Document in MCP instructions and tool description.

| Syntax | Example | Meaning |
|--------|---------|---------|
| Keywords | `heartbeat timer` | Both must match (implicit AND) |
| Phrase | `"systemd timer"` | Exact phrase |
| Boolean | `heartbeat AND NOT clawd` | Boolean operators |
| Prefix | `buffer*` | Prefix matching |
| OR | `heartbeat OR cron` | Either term |
| Grouping | `(timer OR cron) AND heartbeat` | Grouped boolean |

**Behavior change from current:** Default query behavior switches from implicit OR
(bm25s scores both terms independently) to implicit AND (FTS5 requires all terms).
This gives more precise results. Users wanting OR behavior use explicit `OR`.

Document this in the MCP server instructions and the `search_conversations` tool docstring.

### Highlighted Snippets

Replace truncated first-300-chars with FTS5 `snippet()`:

```
Current:  "This is the first 300 characters of the turn text regardless of where..."
New:      "...the [[heartbeat]] system uses a [[timer]] with 30-minute intervals..."
```

`snippet(turns_fts, 0, '[[', ']]', '...', 24)` produces context windows with
`[[bracket]]` markers around matching terms. Column 0 = `text`. Max 24 tokens of context.

**Why `[[`/`]]` instead of `**`:** The search results are JSON consumed by LLMs via MCP.
Markdown bold markers can collide with actual markdown/code content in the indexed text.
`[[`/`]]` are compact, ASCII, readable, and not tied to any rendering format.

## Full-Fidelity Reads

`read_turn` and `read_conversation` continue to re-parse the source JSONL file.
No change here — these tools need complete text and rendered tool details that
the FTS5 index intentionally doesn't store.

instead of the in-memory dict:

def read_turn(self, session_id, turn_number):

## Daemon Mode Changes

SQLite persistence makes the daemon strictly better:

| Aspect | Before (bm25s) | After (FTS5) |
|--------|----------------|--------------|
| Cold start | Full parse + index (~10s) | Open DB + incremental (~1s) |
| Memory | ~250 MB in-memory index | ~10 MB (SQLite paging) |
| Restart | Full rebuild | Incremental only |
| Multiple sessions | Need daemon to share index | DB file shareable (WAL mode) |

The `connect` bridge architecture stays the same. The daemon still provides:
- Filesystem watching for live reindex
- SSE transport for multi-session sharing
- Idle timeout for resource management

**New possibility (future):** With a persistent DB, the `serve` (stdio) mode could
skip the daemon entirely and query the DB directly. This would make daemon mode
optional rather than recommended for multi-session use. Not in scope for v1, but
the architecture enables it.

## SQLite Configuration

```python
conn = sqlite3.connect(str(DB_PATH), check_same_thread=False)
conn.execute("PRAGMA journal_mode=WAL")     # concurrent reads
conn.execute("PRAGMA synchronous=NORMAL")   # faster writes, safe with WAL
conn.execute("PRAGMA cache_size=-8000")      # 8MB page cache
```

WAL mode is critical — allows the daemon to write while CLI/stdio instances read.
WAL provides **cross-process** concurrency (daemon writes, CLI reads). It does NOT
make a single shared connection thread-safe.

## Threading Model

Use a single `threading.Lock` that covers all DB operations end-to-end. The lock
must protect the full operation (acquire → execute SQL → process results → release),
not just connection acquisition.

```python
class ConversationIndex:
    def __init__(self, ...):
        self._lock = threading.Lock()

    def search(self, query, ...):
        with self._lock:  # covers entire operation
            conn = self._get_connection()
            rows = conn.execute(sql, params).fetchall()
            # ... process rows ...
            return results

    def build(self, pattern):
        with self._lock:  # covers entire build
            conn = self._get_connection()
            # ... all inserts/deletes ...
            conn.commit()
```

This is sufficient for our workload: low concurrency, fast operations. The daemon's
filesystem watcher triggers `build()` and request handlers call `search()` — the lock
serializes these safely.

## ConversationIndex Refactor

The class interface is unchanged. Internal changes:

```python
class ConversationIndex:
    def __init__(self, db_path: Path = DB_PATH):
        self._db_path = db_path
        self._conn: sqlite3.Connection | None = None
        self._lock = threading.Lock()  # keep for thread safety

    def _get_connection(self) -> sqlite3.Connection:
        if self._conn is None:
            self._conn = self._open_db()
        return self._conn

    def _open_db(self) -> sqlite3.Connection:
        self._db_path.parent.mkdir(parents=True, exist_ok=True)
        conn = sqlite3.connect(str(self._db_path), check_same_thread=False)
        conn.execute("PRAGMA journal_mode=WAL")
        conn.execute("PRAGMA synchronous=NORMAL")
        _create_schema(conn)
        return conn

    def build(self, pattern: str) -> None: ...     # incremental, writes to DB
    def search(self, ...) -> dict: ...              # FTS5 MATCH + recency
    def list_conversations(self, ...) -> dict: ...  # SQL query on sessions
    def read_turn(self, ...) -> dict: ...           # file_path from sessions table
    def read_conversation(self, ...) -> dict: ...   # same
```

**Removed:**
- `self._retriever` (bm25s.BM25 instance)
- `self._corpus` (list of turn dicts)
- `self._conversations` (dict of session metadata)
- `self._session_files` (dict of session → path)
- `self._file_cache` (incremental cache — now in SQLite)

All of these are replaced by SQLite queries against `sessions` and `turns_fts` tables.

## Error Handling

### FTS5 Query Syntax Errors

FTS5 query syntax errors (e.g., unmatched quotes, invalid boolean, code-like queries
with special characters) raise `sqlite3.OperationalError`.

**Strategy: try raw, fallback on error, explicit escape hatch.**

```python
def _execute_fts_query(self, conn, sql, params, query):
    """Execute FTS5 query with automatic fallback for syntax errors."""
    try:
        return conn.execute(sql, params).fetchall()
    except sqlite3.OperationalError as e:
        if "fts5" not in str(e).lower():
            raise
        # Fallback: extract tokens and retry as quoted phrase
        tokens = _extract_search_tokens(query)
        if not tokens:
            return []
        fallback_query = '"' + " ".join(tokens) + '"'
        fallback_params = list(params)
        fallback_params[0] = fallback_query
        try:
            return conn.execute(sql, fallback_params).fetchall()
        except sqlite3.OperationalError:
            return []  # give up gracefully

def _extract_search_tokens(query: str) -> list[str]:
    """Extract FTS-compatible tokens from a query string.
    Strips punctuation/operators that break FTS5 syntax."""
    # Remove FTS5 operators and special chars, keep alphanumeric + underscore
    import re
    return re.findall(r'\w+', query)
```

**`literal:` prefix:** Users can prefix queries with `literal:` to skip FTS5 syntax
parsing entirely. The prefix is stripped and the remainder is tokenized and searched
as a quoted phrase. This provides a deterministic escape hatch for code-like queries.

```python
if query.startswith("literal:"):
    raw = query[len("literal:"):]
    tokens = _extract_search_tokens(raw)
    query = '"' + " ".join(tokens) + '"' if tokens else raw
```

### FTS5 Availability Check

On startup, verify FTS5 is compiled into the SQLite library:

```python
def _check_fts5_available(conn):
    try:
        conn.execute("CREATE VIRTUAL TABLE _fts5_check USING fts5(x)")
        conn.execute("DROP TABLE _fts5_check")
    except sqlite3.OperationalError:
        raise RuntimeError(
            "SQLite FTS5 extension not available. "
            "Python 3.10+ on Linux/macOS should include it. "
            "Check your Python/SQLite build."
        )
```

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| FTS5 not compiled into Python's sqlite3 | Startup probe (see §Error Handling). Python 3.10+ on Linux/macOS ships with FTS5. Verified on Archibald (SQLite 3.51.2). |
| Implicit AND breaks existing search habits | Document clearly. OR is explicit. AND is actually better for precision. |
| DB corruption / schema drift | `PRAGMA user_version` check on startup. Mismatch → drop + rebuild. DB is a cache; JSONL is source of truth. |
| Concurrent write conflicts | Single writer (daemon or CLI). SQLite WAL for cross-process readers. `threading.Lock` for intra-process safety. |
| Index size on disk | ~3000 turns × ~2KB avg = ~6MB FTS5 index. Negligible. |
| Code-like queries breaking FTS5 syntax | Try raw → fallback to tokenized phrase on error. `literal:` prefix for explicit escape. See §Error Handling. |
| Pattern-scoped stale deletion | Stale check scoped to current glob pattern only. Rows from other patterns are never touched. |
| Thread safety with shared connection | Lock covers full DB operation, not just connection acquisition. See §Threading Model. |

## Out of Scope (v1)

- Column-targeted search (`user_text:query` vs `assistant_text:query`) — possible with separate FTS5 columns, but adds complexity. Revisit if requested.
- Storing full turn text in DB (skip JSONL re-parse for reads) — trades disk for speed. Not needed at current scale.
- Codex session support — trivial to add (recall has the parser), but no immediate need.
- Direct DB access from stdio mode (bypass daemon) — architecturally possible, defer.
